# Research: PanVK upstream status for Mali-G52 (Bifrost v7) on RK3566 / RG353M / Odin Lite

## Summary
PanVK in Mesa 25.2.6 (and every released branch since) is conformant only on Mali-G610 (Valhall CSF, arch v10); Mali-G52 (Bifrost v7) is one of the architectures Collabora explicitly lists as "completely lacking support," which is why `PAN_I_WANT_A_BROKEN_VULKAN_DRIVER=1` lets the ICD load but `vkEnumeratePhysicalDevices` returns 0 devices with `VK_ERROR_INCOMPATIBLE_DRIVER`. No merged or in-flight Mesa merge request bringing Bifrost v7 up to enumeration parity exists in 25.3 / 26.0 / 26.1 / 26.2; the visible PanVK roadmap energy is going into Valhall/CSF (v10–v14) and into extensions on already-supported arches. Practically, "wait for upstream PanVK" is not a viable path on a sub-12-month horizon for Mali-G52.

## Findings

1. **PanVK is conformant only on Mali-G610; G52 is officially non-conformant.** The upstream Mesa docs page states: "PanVK, the Vulkan implementation in the Panfrost driver stack, is currently conformant on Mali-G610, but non-conformant on other GPUs." Panfrost (OpenGL ES) is conformant on G52, but the Vulkan driver is not. [docs.mesa3d.org/drivers/panfrost.html](https://docs.mesa3d.org/drivers/panfrost.html)

2. **Bifrost v6/v7 and Valhall v9 are explicitly listed as unsupported in PanVK.** Collabora's Mesa 25.2 / Vulkan 1.4 announcement: "we have older GPU generations that we'd like to bring up to speed, like Bifrost (V6 and V7), plus the first generation of Valhall (V9) which is currently completely lacking support." Mali-G52 is Bifrost v7 — i.e. the driver does not have a per-arch backend (`panvk_vX_*` for X=7) capable of completing physical device enumeration. [collabora.com – PanVK now supports Vulkan 1.4](https://www.collabora.com/news-and-blog/news-and-events/panvk-now-supports-vulkan-1.4.html)

3. **The `PAN_I_WANT_A_BROKEN_VULKAN_DRIVER=1` gate is the project-wide non-conformant escape hatch.** Phoronix's coverage of the Mesa 26.2 Mali-G1-Pro (arch v14) merge explicitly confirms the env var is still how PanVK exposes any not-yet-conformant arch — i.e. enabling the env var only matters when there is a per-arch backend; for v7 it lets the loader try and then fail in `panvk_physical_device.c` because no arch dispatch matches. [phoronix.com – Arm Mali G1 Pro Now Working With PanVK & Panfrost](https://www.phoronix.com/news/Arm-Mali-G1-Pro-Mesa-26.2)

4. **PanVK uses a compile-time per-architecture dispatch (`PAN_ARCH`).** DeepWiki's summary of `src/panfrost/vulkan/`: "PanVK implements the Vulkan API for ARM Mali GPUs across multiple hardware generations. The driver uses a compile-time architecture dispatch mechanism (PAN_ARCH) to handle generation-specific differences while sharing common infrastructure." `panvk_physical_device.c` matches the detected `gpu_id` to a compiled-in arch table; when v7 is not compiled (the meson build for PanVK only lists v6, v10, v12, v13, plus v14 in 26.2 — not v7 or v9), the create-physical-device path bails with `VK_ERROR_INCOMPATIBLE_DRIVER` after printing the "panvk is not well-tested" warning. Adding v7 would require a real per-arch backend (descriptors, format table, queue, shader compile path), not a one-line allow-list. [deepwiki.com – PanVK ARM Mali Vulkan Driver](https://deepwiki.com/bminor/mesa-mesa/2.4-panvk-arm-mali-vulkan-driver)

5. **Bifrost compiler does cover v6/v7 — the gap is the Vulkan frontend, not the shader compiler.** "bifrost_compile.c … supports Bifrost (v6, v7) and Valhall (v9+)." This is why Panfrost (GLES) works on G52 but PanVK does not: the NIR→Bifrost compiler is there, but PanVK's descriptors, queue, image, and physical-device layers are not specialized for v7. [deepwiki.com – Panfrost and PanVK Arm Mali Drivers](https://deepwiki.com/arehnman/virtio-win-mesa/7.1-panfrost-and-panvk:-arm-mali-drivers)

6. **Mesa 25.3 / 26.0 / 26.1 / 26.2 release-note evidence shows zero Bifrost v7 PanVK work.** The 26.0 notes list panvk changes scoped to "v10+", AFBC tiling, bifrost disassembly fix-ups — nothing that adds a v7 backend ([Mesa 26.0.0 relnotes](https://docs.mesa3d.org/relnotes/26.0.0.html)). 26.1 is Christian Gmeiner's "extension sprint" on already-supported arches and a v9+-only ASTC feature, with no v7 enablement ([christian-gmeiner.info – PanVK Extension Sprint Mesa 26.1](https://christian-gmeiner.info/2026-04-20-panvk-extensions/)). 26.2 adds a brand-new arch (v14, Mali G1-Pro / 5th-gen Valhall) but still gated behind the broken-driver env var ([Phoronix](https://www.phoronix.com/news/Arm-Mali-G1-Pro-Mesa-26.2)). The trajectory is forward (newer Valhall/CSF), not backward to Bifrost gen 2/3.

7. **Mesa 25.1 (April 2025) added Mali 5th-gen gen1 (v12) and gen2 (v13) to Panfrost/PanVK, again skipping v7/v9.** Phoronix: "the Panfrost Gallium3D and PanVK Vulkan drivers for Arm Mali open-source graphics are supporting Mali 5th Gen gen 1 (v12) and gen 2 (v13) devices." The pattern over a year of releases is: invest in the cores Collabora's commercial customers care about (CSF / Valhall), defer Bifrost v6/v7. [phoronix.com – Mesa 25.1 Newer Mali 5th Gen](https://www.phoronix.com/news/Mesa-25.1-Newer-Mali-5th-Gen)

8. **gamescope-required extensions on PanVK — partial picture.**
   - `VK_KHR_wayland_surface`: PanVK has a WSI/Wayland surface code path; Mesa 26.1 added related surface-capabilities extensions on PanVK (`VK_KHR_get_surface_capabilities2`, `VK_KHR_get_display_properties2`, `VK_EXT_acquire_drm_display`, `VK_KHR_present_id`, `VK_KHR_present_wait`), which presupposes a working surface. [phoronix.com – Mesa 26.1-rc1](https://www.phoronix.com/news/Mesa-26.1-rc1-Released)
   - `VK_EXT_physical_device_drm`: confirmed exposed (PanVK calls `pan_kmod_dev_create` and identifies the DRM node; DRM-display/acquire extensions in 26.1 build on it). [deepwiki](https://deepwiki.com/arehnman/virtio-win-mesa/7.1-panfrost-and-panvk:-arm-mali-drivers)
   - `VK_EXT_image_drm_format_modifier` (DMA-BUF import/export with modifiers): present on the conformant Valhall/CSF path; 26.0 added explicit `get_dmabuf_modifier_planes` overrides and 422 AFBC modifier formats for Panfrost. PVR landed `VK_EXT_image_drm_format_modifier` only in 26.1; PanVK's modifier story is tied to AFBC/U-interleaved tilings exposed per-arch — a v7 backend would have to re-derive the modifier list. [Mesa 26.0 relnotes](https://docs.mesa3d.org/relnotes/26.0.0.html), [Mesa 26.1 relnotes](https://docs.mesa3d.org/relnotes/26.1.0.html)
   - Bottom line: on the conformant G610 path PanVK can satisfy gamescope's hard checks; on G52 the question is moot because no `VkPhysicalDevice` is enumerated.

9. **Honeykrisp (Apple M1 Vulkan) is *not* a PanVK fork.** Phoronix and Asahi sources are explicit: Honeykrisp is "derived from the NVK driver," merged into Mesa 24.3 as a new Vulkan driver for Apple Silicon. There is no shared Bifrost code path with PanVK; Honeykrisp progress provides no read-across or upstream pressure for Mali-G52. The premise of "Asahi forked PanVK and that's why Linux PanVK is stalled" is incorrect. [phoronix.com – Honeykrisp derived from NVK](https://www.phoronix.com/forums/forum/linux-graphics-x-org-drivers/vulkan/1469110-honeykrisp-is-a-new-vulkan-driver-for-apple-m1-on-linux-derived-from-the-nvk-driver), [phoronix.com – Apple Vulkan Driver Mesa 24.3](https://www.phoronix.com/news/Apple-Vulkan-Driver-Mesa-24.3)

10. **Community confirmation: G52 PanVK still effectively unusable on RK3566.** OrangePi 3B user porting PanVK to Android 15: "Since rk3566 has mali g52 gpu, it is only vulkan 1.0 conformant" — and even that required out-of-tree work, not stock Mesa. Reports for ROCKNIX/Manjaro/Librem 5 all show the same `PAN_I_WANT_A_BROKEN_VULKAN_DRIVER` + warning + `VK_ERROR_INCOMPATIBLE_DRIVER` chain. [reddit.com/r/OrangePI PanVK on Android 15 orangepi 3b](https://www.reddit.com/r/OrangePI/comments/1km80i8/panvk_panfrosts_vulkan_on_android_15_orangepi_3b/), [forums.puri.sm Broken Vulkan driver on Librem 5](https://forums.puri.sm/t/broken-vulkan-driver-on-librem-5/25325)

11. **ROCKNIX's recent "Vulkan on RK3566" release is NOT PanVK enablement.** The ROCKNIX nightly that "adds Vulkan to RK3566 handhelds. Incredible boost to PSP" refers to PPSSPP's Vulkan backend riding on whatever stack is wired up — but tracking the corresponding work item (`distribution#656`) shows it is **RK3588 Mali blob + Wayland WSI**, not an RK3566 PanVK patch. There is no ROCKNIX, JELOS, or Batocera tree that patches PanVK to enumerate Mali-G52; none of them backport a v7 backend that doesn't exist upstream. [github.com/ROCKNIX/distribution PR #656 "RK3588 vulkan"](https://github.com/ROCKNIX/distribution/pull/656), [forum.odroid.com t=48978](https://forum.odroid.com/viewtopic.php?t=48978)

12. **Honest 12-month assessment.** A v7 PanVK backend requires (a) panvk_v7_physical_device with format/feature tables, (b) panvk_v7_queue and JM job submission specifics, (c) descriptor + image + buffer layout code (Bifrost gen 2 layout differs from v6 and from Valhall), and (d) CTS bring-up to clear conformance. Collabora's public statement frames v6/v7/v9 as wishlist items with no commitment. Mesa's six-month cadence means the earliest plausible appearance of even an experimental v7 enumeration backend is Mesa 26.3 (mid-2026) and more realistically beyond — and even then it would be gated behind `PAN_I_WANT_A_BROKEN_VULKAN_DRIVER=1` for many more cycles before it could pretend to satisfy a gamescope-class consumer. Planning around upstream PanVK for RG353M / Odin Lite within 12 months is not credible; the practical levers are the Mali blob (RK3588-style WSI hack) or CPU/llvmpipe.
